# Agent Backpressure

## What Backpressure Means Here

In agent-driven development, backpressure is the resistance signal that tells an
agent its output was wrong and exactly how to fix it. The term is borrowed
deliberately: just as hydraulic backpressure resists flow until upstream pressure
builds to push through, a well-designed feedback system resists bad output until
the agent corrects it.

**Your backpressure as an engineer is finite.** Every time you type a message
telling an agent it missed an import, forgot a semicolon, or broke a layout, you
are spending it. That is time taken away from architecture, judgment, and problems
machines cannot solve. Any feedback a machine can generate automatically is
feedback you should never be giving manually.

The central question to ask about any agent workflow: *am I wasting my
backpressure?*

---

## The Four Properties of Good Backpressure

Feedback signals that work share these properties. Use them to evaluate any check
in your pipeline:

| Property | What it means | What bad looks like |
|----------|--------------|---------------------|
| **Automated** | Generated by a machine, not you | You are typing the same correction more than once |
| **Immediate** | Fires at the moment of the mistake, not later | Agent commits broken code; you catch it on review |
| **Specific** | Names the file, line, and reason | "Build failed" with no further detail |
| **Actionable** | The agent can fix it without asking you | Vague lint warnings the agent ignores |

When a check fails all four, remove it or improve it. A check that is automated
but not specific (e.g. "tests failed" with no output) is worse than useless — it
consumes the agent's context without providing a path forward.

## Backpressure Layers

Think in layers. Each layer catches a different class of mistake. The goal is
full coverage with no gaps that require you to fill in manually.

### Layer 1: Compiler / Type System

The fastest and cheapest feedback. Fires before any code runs.

- **Prefer languages with expressive type systems.** TypeScript, Rust, Elm, and
  even modern Python with strict type checking catch whole classes of mistakes
  before runtime. A TypeScript compiler error that says "Property 'userId' does
  not exist on type 'Session'" is more actionable than any test failure.
- **Rust and Elm are praised specifically** for error message quality — they
  suggest fixes, not just locations. This matters because the error is fed
  directly back into the agent's context window.
- **Prefer high-performance type checkers.** Slow typechecks create pressure to
  skip them. For Python, prefer **`pyright`** (used by Pylance, sub-second on
  large codebases) over mypy. For Python dependency and environment management,
  prefer **`uv`** — it replaces pip, pip-tools, virtualenv, and pyenv with a
  single Rust-backed tool that resolves and installs orders of magnitude faster.
  Faster tooling means the gate runs in seconds, not minutes.
- **Schema-generated types close the gap between DB and code.** Prisma is a
  strong example: migrations generate TypeScript types, so a schema change that
  breaks a query fails at typecheck, not at runtime. Auto-generating types from
  schemas eliminates an entire category of drift.
- Run typechecks on the **whole project**, not just changed files. Agents make
  changes that break distant call sites.

### Layer 2: Linter

Catches style, pattern, and common logic errors. Must be zero-tolerance.

- **Zero warnings policy.** A linter set to "warn" on anything is a linter the
  agent learns to ignore. All rules are either errors or removed.
- **Prefer high-performance linters and formatters.** Tool speed directly affects
  whether agents bother looping. For Python, **`ruff`** replaces flake8, isort,
  pyupgrade, and black in a single Rust-backed binary that runs 10–100x faster
  than the tools it replaces — fast enough to run on every save, not just on
  commit. For JavaScript/TypeScript, **`oxc`** (oxidation compiler) and
  **`Biome`** offer similar Rust-backed performance over eslint and prettier.
  Prefer these over legacy toolchains on any new project.
- **Auto-fix on staged files before commit.** Format and lint issues that can be
  automatically fixed should never reach the agent as feedback — they should
  just be fixed. Only unfixable issues should block.
- **LSP-backed linters via MCP** extend this layer into the agent's context
  during inference, not just at commit time. If your stack has an LSP, attach it.

### Layer 3: Unit Tests

Catches logic errors in isolated units. Must be fast.

- **Tests are backpressure the agent generates for itself.** When an agent writes
  a unit test for the code it just produced, it creates a feedback loop it owns.
  Encourage this pattern explicitly in your CLAUDE.md.
- **Fast suite = used suite.** If unit tests take more than 30 seconds, agents
  skip them mentally. Keep them under 10 seconds.
- **`--passWithNoTests` at bootstrap.** Empty suites must not fail. The pipeline
  must be green before tests exist.

### Layer 4: End-to-End Tests

Catches integration failures across the full stack.

- **Playwright with MCP integration is the current gold standard** for UI
  feedback. The agent makes a change, renders the page in a headed browser via
  the Playwright MCP server, and compares what it sees against expectation —
  without asking you whether the layout looks right.
- **Service-specific E2E via API.** Services like Mailpit expose HTTP APIs. An
  agent testing an email flow can query Mailpit directly rather than asking you
  to check your inbox. Wire these up.
- **E2E is slow — gate it.** E2E runs on commit, not on every file save. The
  pre-commit hook should check the dev server is reachable before running E2E,
  and provide a clear remedy command if it isn't.

### Layer 5: MCP Context Tools

Not feedback in the traditional sense — but prevents the agent from needing to
ask you for context it could retrieve itself.

| MCP Server | What it removes from your plate |
|------------|----------------------------------|
| `playwright` / Chrome DevTools | You manually describing UI state |
| `postgres` | You manually checking DB state or query results |
| `filesystem` | You manually telling the agent what files exist |
| `docker` | You manually checking container health |
| Service APIs (Mailpit, etc.) | You manually verifying external service state |

**Rule:** if you find yourself describing a system state to an agent that a tool
could retrieve, add the tool. That description is wasted backpressure.

If a new technology is introduced to the project, check whether it has an MCP
integration before integrating it any other way.

## Stack Selection Principles

When choosing technologies for an agent-driven project, weight these properties
in order:

1. **Type safety at compile time.** Prefer TypeScript over JavaScript, Rust over
   C, strict type checking over untyped Python. The compiler is your fastest
   feedback layer and it costs nothing to run. For Python, prefer `pyright` over
   `mypy` for speed; manage the environment with `uv` over pip.

2. **Schema-driven type generation.** Any boundary between services (DB, API,
   queue) should generate types automatically. Prisma + PostgreSQL is a strong
   default for relational data. OpenAPI codegen for HTTP boundaries. The goal is
   zero hand-written type definitions at service boundaries.

3. **Agent familiarity.** Choose technologies the model has seen extensively in
   training. Keycloak for auth, Next.js for fullstack, Jest for unit tests,
   Playwright for E2E. Obscure or very new frameworks reduce the quality of agent
   output because the model has less to draw on.

4. **MCP coverage.** Prefer technologies with existing MCP servers. A DB with an
   MCP server is worth more to an agent than a slightly faster DB without one.

5. **Error message quality.** Given two equivalent tools, prefer the one with
   better error messages. This is not a minor consideration — the error message
   IS the backpressure.

## The Ralph Loop Pattern

Agent workflows that run autonomously operate as loops: the agent takes an action,
receives feedback, corrects, and repeats until all checks pass. This is sometimes
called a "ralph loop" or "agent harness" pattern.

Your backpressure infrastructure is what makes this loop converge rather than
spiral. For the loop to be self-healing:

- **Every failure must be recoverable by the agent without you.** If any check
  produces output the agent cannot act on alone, the loop stalls and you get
  paged. Audit your error messages with this lens.
- **Specs and plans front-load constraints.** Requirements given during the
  planning phase ("spec" or "plan" phase) are cheaper than requirements discovered
  during execution. Write your constraints before the loop starts, not as
  corrections mid-loop.
- **Agents should generate their own checks.** An agent that writes unit tests,
  adds type annotations, and creates E2E scenarios as part of its output is
  building backpressure infrastructure for the next agent that touches that code.
  This compounds.
- **Destructive operations need safe sandboxes.** Agents should be able to
  observe consequences of destructive actions (dropping tables, resetting state)
  without affecting shared environments. Docker volumes that reset cleanly
  (`rm -rf .docker/volumes/`) and isolated test databases serve this purpose.

## Diagnosing Wasted Backpressure

If you find yourself repeatedly giving an agent the same class of correction,
work through this checklist:

| Symptom | Likely cause | Fix |
|---------|-------------|-----|
| Agent misses imports repeatedly | No typecheck in pre-commit | Add `just typecheck` to the gate |
| Agent breaks existing tests | Tests not running on commit | Add `just test` to the gate |
| Agent produces wrong UI layout | No visual feedback tool | Attach Playwright MCP server |
| Agent uses stale DB schema | No schema-generated types | Add Prisma or equivalent codegen |
| Agent asks about container state | No docker MCP | Attach docker MCP or add health check output to error messages |
| Agent commits code that doesn't compile | Compiler not in pre-commit | Compiler check must be in the gate, not optional |
| Agent ignores lint warnings | Warnings not treated as errors | Set all lint rules to error; remove rules you don't enforce |
| Agent drifts across long sessions | No structured CLAUDE.md | Write explicit verification table; agent re-reads it each session |
